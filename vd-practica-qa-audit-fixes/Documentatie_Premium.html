VreauDigitalizare

Analiză de profunzime (Deep Dive) a ecosistemului VreauDigitalizare

Prezentul document tehnic extins oferă o vizibilitate de 100% asupra mecanismelor interne ale platformei VreauDigitalizare. Este conceput pentru a satisface rigorile oricărui audit academic sau tehnic (Cod, Bază de Date, State Management, Performanță, Securitate).

1. Paradigma Arhitecturală și Randarea

1.1. Next.js 14 App Router vs SPA

Aplicațiile tradiționale React (ex: create-react-app) folosesc CSR (Client-Side Rendering). Asta înseamnă că serverul trimite un fișier HTML gol și un fișier JS masiv. Dispozitivul utilizatorului trebuie să descarce, să parseze și să execute acel JS pentru a "desena" (randa) pagina. Acest lucru afectează drastic timpul de încărcare pe telefoanele slabe și anulează complet indexarea SEO inițială.

Pentru VreauDigitalizare, am implementat Next.js 14 cu App Router, un framework Full-Stack bazat pe paradigma React Server Components (RSC). RSC mută "greutatea" pe server. Când un utilizator vizitează site-ul:

  1. Serverul Vercel preia request-ul HTTP.
  2. Execută interogările la baza de date (Prisma) direct în componenta React de pe server (Fără fetch APIs interne).
  3. Compilează HTML-ul complet formatat și îl trimite instant clientului (Time To First Byte - TTFB < 50ms).
  4. Trimite un pachet minim de JavaScript doar pentru Hidratare (ex: pentru a face butonul de Chat interactiv).

1.2. State Management (Gestionarea Stării)

Managementul stării este hibrid, adaptat strict la necesitate (fără a supraîncărca aplicația cu Redux):

  • Starea Serverului (Data State): Gestionată direct prin Next.js Caching și Server Actions. După o mutație (ex: crearea unui lead), apelăm revalidatePath(), care actualizează automat starea paginii pe tot parcursul aplicației.
  • Starea Formularelor (Form State): Gestionată local folosind React Hook Form. Oferă validare în timp real fără să forțeze o re-randare a întregului DOM la fiecare tastare (performanță mult superioară form-urilor controlate tradițional).
  • Starea UI Globală (Theme / Sidebar): Gestionată prin React Context API (ex: ThemeProvider) și useState / useEffect combinate cu LocalStorage.

2. Schema Relațională a Bazei de Date

Platforma utilizează PostgreSQL, interogat printr-un ORM puternic tipizat: Prisma. ORM-ul asigură că nu putem scrie date eronate în baza de date (Type-Safety absolut din Frontend până în Backend).

2.1. Modelarea Relațiilor de Business

Sistemul de CRM (Lead Management) a fost modelat folosind relații "One-to-Many". Iată fragmentul exact din schema.prisma care explică legătura dintre un prospect și auditul său:

model Lead { id String @id @default(cuid()) name String email String @unique company String? status LeadStatus @default(NEW) score Int? notes String? createdAt DateTime @default(now()) // Relație 1-to-Many cu Răspunsurile responses Response[] } model Response { id String @id @default(cuid()) leadId String questionId String answerId String // Constrângeri Foreign Key lead Lead @relation(fields: [leadId], references: [id], onDelete: Cascade) }

Decizie Tehnică: Am folosit onDelete: Cascade. Asta înseamnă că dacă administratorul șterge un Lead din panoul de bord, PostgreSQL va curăța automat și toate răspunsurile acestuia din tabela Response, prevenind anomaliile de baze de date ("Orphaned records").

3. Disecția Componentelor de Frontend (Deep Dive)

Componenta WebGL: LiquidEther.tsx

Majoritatea site-urilor folosesc clipuri video pe fundal, care consumă zeci de MB de lățime de bandă. Noi am optat pentru randare GPU în timp real.

Mecanism de funcționare: Instanțiem o pânză <canvas> HTML5. Folosind GLSL (OpenGL Shading Language), scriem două programe (Vertex Shader și Fragment Shader) care calculează dinamica fluidelor bazate pe ecuațiile Navier-Stokes. Interacțiunea (pointerul mouse-ului) este mapată pe coordonatele canvas-ului cu getBoundingClientRect(), transformată în presiune (velocity) și injectată în simularea shader-ului de densitate.

Sistemul de Rute CMS: [slug]/page.tsx

Blogul nu folosește ID-uri urâte de tipul /blog/12345, ci URL-uri Semantice (Slugs): /blog/ce-este-digitalizarea.

Mecanism de funcționare: Pagina folosește "Dynamic Routing" oferit de Next.js. Funcția generateStaticParams() este chemată la build-time. Aceasta interoghează baza de date pentru toate articolele publicate și pre-generează HTML-ul static pentru fiecare dintre ele. Astfel, când un vizitator accesează un articol, timpul de încărcare este de ~30ms, oferind scor 100/100 la Lighthouse (SEO Perfect).

4. Backend și Prevenirea Erorilor (Zod & Server Actions)

4.1. Server Actions vs REST API

Dezvoltarea tradițională implică crearea unui fișier controller separat (ex: în Express) și scrierea unui fetch cu JSON.stringify pe frontend. Cu React 19 / Next.js 14 Server Actions, mutațiile sunt declarate direct ca funcții asincrone pe server.

// actions.ts (Rulează STRICT pe server) "use server"; import { z } from "zod"; import { prisma } from "@/lib/db"; // Schema de securitate/validare const formSchema = z.object({ email: z.string().email(), name: z.string().min(2) }); export async function submitLead(formData: FormData) { // 1. Extragere Date const rawData = Object.fromEntries(formData); // 2. Validare Zod (Respinge atacurile) const validated = formSchema.safeParse(rawData); if (!validated.success) throw new Error("Invalid Data"); // 3. Salvare Sigură const lead = await prisma.lead.create({ data: validated.data }); return { success: true, id: lead.id }; }

4.2. Securitatea Formularelor (Biblioteca Zod)

Nicio cerere HTTP nu atinge baza de date fără a trece prin "Vama" Zod. Zod verifică lungimea string-urilor, formatele de email, tipurile datelor. Previne complet atacurile de tip SQL Injection (combinat cu parameterized queries ale Prisma) și Mass Assignment, acceptând în obiectul data strict câmpurile definite în schemă.

5. Algoritmul de Generare al Sloturilor (Booking)

Aplicația include un sistem de agendă inteligentă. Provocarea tehnică a fost ca baza de date PostgreSQL să nu accepte niciodată o suprapunere a două ședințe pentru același interval orar.

Cum funcționează algoritmul matematic intern:

  1. Ruta /api/booking/available-slots extrage parametrul din URL: req.nextUrl.searchParams.get("date").
  2. Datorită interogărilor dinamice (req.url), fișierul are definit obligatoriu export const dynamic = 'force-dynamic'. (În caz contrar, compilatorul Vercel ar arunca eroarea "Dynamic Server Usage" și ar opri deployment-ul).
  3. Generarea matricei de bază: Presupunem program 09:00 - 13:00. Se instanțiază obiecte Date() din 30 în 30 de minute (sau conform setărilor din tabela CalendarSettings). Rezultă array-ul: ['09:00', '09:30', ... '12:30'].
  4. Coliziunea cu Appointments: Prisma extrage din tabela Appointment toate rândurile unde data corespunde și status IN ('APPROVED', 'PENDING').
  5. Se rulează metoda Array.prototype.filter() pe matricea de bază. Orice slot care este cuprins între startTime și endTime a unei programări găsite, este eliminat iterativ. Slot-urile rămase sunt trimise curat, via JSON, către calendarul din UI (Frontend).

6. Design System și Optimizări CSS (Tailwind)

Platforma nu folosește CSS Vanilla dezordonat. A fost creat un sistem de Design Tokens (Sistemul "Obsidian & Rose").

  • Configurarea s-a realizat direct în tailwind.config.ts, unde am definit paleta de culori în secțiunea theme.extend.colors (ex: primary: "var(--primary)" unde primary este #DB7F8E).
  • Framer Motion: A fost adăugat pentru a prelua controlul asupra DOM-ului virtual. Animațiile (ex: tranziția întrebărilor din chestionar) se execută pe GPU (folosind proprietăți de transform și opacity) fără a afecta CPU-ul și evitând fenomenul de "layout shift" (CLS).

7. Deployment Pipeline, CI/CD și Securitate (Vercel)

Mediul de producție a fost complet automatizat pe infrastructura Vercel (Serverless Edge Network).

  • CI/CD (Continuous Integration): La fiecare git push pe branch-ul main, Vercel descarcă codul, verifică dependențele (npm/pnpm), rulează tsc (TypeScript Compiler) pentru a asigura că nu există nicio eroare de tip (Type Error), compilează paginile statice și pune aplicația online în mai puțin de 2 minute.
  • Variables Environment: Cheile private (precum DATABASE_URL spre PostgreSQL sau AUTH_SECRET folosit pentru criptarea sesiunilor JWT în NextAuth) sunt stocate criptat în seiful Vercel. S-a folosit biblioteca @t3-oss/env-nextjs pentru a garanta, la momentul compilării (Build Time), că nicio variabilă obligatorie nu lipsește.
  • NextAuth (JWT): Sesiunile administratorilor nu sunt salvate fizic pe server, făcând serverul "stateless". După login, aplicația generează un JWE (JSON Web Encryption) Token salvat securizat (HttpOnly) în cookie-ul browser-ului. La fiecare request pe /admin, Middleware-ul Edge Next.js decodează token-ul și validează accesul în 10 milisecunde.
