Această documentație detaliată servește drept resursă definitivă (Single Source of Truth) pentru dezvoltatorii, inginerii DevOps și consultanții UX/UI care lucrează la platforma VreauDigitalizare. 1. Viziunea Proiectului și Abordarea de Design

1.1. De la "Corporate Blue" la "Corporate Elegance (Floral Earth)"

Aplicațiile tradiționale B2B folosesc predominant scheme de culori reci (Dark Mode, nuanțe de gri și albastru marin) bazate pe premisa că "serios înseamnă sobru". Analizând audiența noastră țintă (antreprenori și manageri de IMM-uri aflați la începutul procesului de transformare digitală), am constatat că acest stil rece creează o barieră (intimidare tehnologică).

Decizia de Design: Am migrat întreaga aplicație către un design denumit Floral Earth sau Corporate Elegance.

  • Fundal Principal (Ivory/Alabaster #FDFBF7): O nuanță caldă de alb-fildeș care elimină oboseala oculară specifică albului pur (#FFFFFF), oferind un aer de publicație editorială premium sau agenție de consultanță boutique de top (ex: McKinsey, IDEO).
  • Text Principal (Bole #83513E): Maro închis în loc de negru pur. Oferă lizibilitate excelentă (WCAG AAA contrast), dar este perceput ca fiind mult mai organic și abordabil de către ochiul uman.
  • Accente Strategice (Salmon Pink #F28AA1 & Fawn #E7AE75): Folosite strict pentru elemente de acțiune (CTA-uri), hover states și notificări. Aduc dinamism fără a compromite profesionalismul.

1.2. Interactivitate și Micro-Animații

Am implementat efecte vizuale de înaltă performanță pentru a demonstra competența tehnologică a agenției chiar din homepage:

  • LiquidEther (WebGL): Un fundal fluid, interactiv la mouse, construit cu Three.js. Reprezintă "fluiditatea transformării digitale".
  • Bento Grids: Prezentarea informațiilor (atât în secțiunea de servicii cât și în Dashboard-ul Admin) prin carduri modulare asimetrice. Reduc încărcătura cognitivă.
  • Framer Motion: Toate tranzițiile între pagini și apariția elementelor se fac prin interpolare spring, creând o experiență tactilă (60 FPS constant).

2. Arhitectura Aplicației (Next.js App Router)

Aplicația utilizează paradigma de tip BFF (Backend for Frontend) facilitată nativ de framework-ul Next.js versiunea 14+, folosind noul App Router.

Cum funcționează Rendering-ul Hibrid:

Am împărțit aplicația în Route Groups logice (ex: (marketing), (admin)) pentru a aplica strategii diferite de randare:

  1. app/(marketing): Paginile publice sunt generate static (SSG). Datele dinamice (ex: ultimele articole de blog) sunt fetch-uite folosind Incremental Static Regeneration (ISR), ceea ce garantează un TTFB (Time to First Byte) sub 50ms, esențial pentru SEO și Core Web Vitals.
  2. app/admin: Tot panoul de administrare forțează randarea pe server (SSR) la cerere (force-dynamic) deoarece administratorii trebuie să vadă lead-urile în timp real, iar datele sunt strict confidențiale (nu trebuie stocate în Edge Cache).

3. Stack Tehnologic: Justificări Detaliate

Tehnologie / Librărie Rol în aplicație Motivul selecției (Trade-offs & Avantaje)
Next.js 14 (App Router) Framework Full-Stack React React simplu necesită un backend separat (Express/Nest). Next.js ne permite să scriem backend-ul (Server Actions) direct în aceleași fișiere ca frontend-ul, accelerând dramatic timpul de dezvoltare. Componentele Server (RSC) reduc masiv cantitatea de JavaScript trimisă clientului.
TypeScript (Strict Mode) Limbaj Procesul de evaluare a lead-urilor implică manipularea de structuri JSON complexe (Rapoarte AI, Răspunsuri). Fără validarea statică a tipurilor oferită de TypeScript, aplicația ar fi fost predispusă la erori majore la runtime (ex: undefined is not an object).
Prisma ORM Interfață Bază de Date Alternativa era TypeORM sau SQL brut. Prisma a fost aleasă pentru type-safety-ul end-to-end. Orice modificare în schema bazei de date se propagă imediat ca eroare de tipare în IDE dacă un query nu a fost actualizat, prevenind bug-uri în producție.
PostgreSQL Bază de date relațională Ales în detrimentul MongoDB (NoSQL) deoarece relațiile noastre (Lead → Răspunsuri → Chestionar) sunt strict relaționale și necesită consistență ACID. PostgreSQL suportă și tipul de date JSONB, esențial pentru stocarea rapoartelor AI și a setărilor flexibile.
Vercel AI SDK Integrare LLM (OpenAI) Gestionează perfect conexiunile de tip streaming (Server-Sent Events) dinspre server spre client, permițând afișarea răspunsurilor AI caracter-cu-caracter, fără a bloca UI-ul.
Zod & React Hook Form Validare Formulare Formularele (login, lead capture, chestionare) sunt validate mai întâi pe client (instant feedback), apoi structura Zod este refolosită pe server pentru securitate, implementând principiul DRY (Don't Repeat Yourself).
Shadcn UI & Tailwind CSS Sistem de Componente & Stil Shadcn nu este o librărie instalabilă ca Bootstrap, ci un set de componente accesibile (Radix UI) pe care le deținem fizic în folderul components/ui. Acest lucru ne-a permis să modificăm codul intern al fiecărei componente pentru a se potrivi temei Floral Earth.

4. Schema Bazei de Date (Prisma ORM)

Baza de date a fost arhitecturată pentru a susține scalabilitatea agenției de consultanță, permițând crearea nelimitată de chestionare dinamice.

Explicația Modelului de Date (Design Relațional):

  • Dynamic Schemas: În loc să creăm coloane specifice în tabelul Lead pentru fiecare întrebare, am creat o arhitectură dinamică. Entitatea Question salvează structura întrebării (opțiunile sunt un câmp JSON), iar ResponseAnswer stochează valoarea completată de lead, creând o relație Many-to-Many rezolvată, permițând crearea de noi chestionare din Dashboard fără a modifica schema SQL.
  • Câmpurile JSON (NoSQL in SQL): Câmpurile precum aiReport sau timeline din modelul Lead sunt de tip Json. De ce? Pentru că structura unui raport generat de AI (pros, cons, recomandări) se poate schimba. Folosind JSON în PostgreSQL obținem flexibilitatea NoSQL păstrând integritatea SQL pentru ID-uri.
  • Cascade Deletes: S-a folosit onDelete: Cascade pe tot arborele de chestionare (Questionnaire -> Question -> QuestionRuleGroup). Dacă un admin șterge un chestionar, toate întrebările, regulile logice și opțiunile asociate se șterg automat la nivel de bază de date pentru a preveni "datele orfane" (orphan records) și scurgerile de memorie.

5. Securitate și Fluxul de Autentificare

Aplicația utilizează NextAuth.js (Auth.js v5) combinat cu Middleware-ul Edge oferit de Next.js pentru securizarea rutelor panoului de control.

sequenceDiagram autonumber participant V as Vizitator participant MW as Edge Middleware participant NA as NextAuth (Server) participant DB as PostgreSQL V->>MW: Cerere către /admin/dashboard MW->>MW: Verifică Session Token (JWT în Cookie) alt Token Lipsă sau Invalid MW-->>V: Redirect 302 către /login?callbackUrl=/admin/dashboard else Token Valid MW-->>NA: Passthrough cerere către Server Component NA->>DB: Prisma.user.findUnique(id) (opțional ptr validare rol) DB-->>NA: Returnează detalii User (Role: ADMIN) alt Rol != ADMIN NA-->>V: Eroare 403 / Pagina Access Denied else Rol == ADMIN NA-->>V: Randează 200 OK /admin/dashboard (SSR) end end

Politici de Securitate Implementate:

  1. Edge Middleware Protection: Rutele /admin/* sunt protejate la nivel de Edge (înainte ca cererea să ajungă la serverul Node.js). Acest lucru oprește atacurile de tip brute-force asupra dashboard-ului fără a consuma resurse de procesare server-side (cost-effective pe Vercel).
  2. Stateless Sessions (JWT): Am ales să stocăm sesiunea în cookies criptate JWT (JWE - JSON Web Encryption) în loc de sesiuni stocate în baza de date. Astfel, verificarea sesiunii durează <2ms și nu supraîncarcă PostgreSQL-ul la fiecare navigare între paginile de admin.
  3. Criptare Parole: Parolele sunt trecute printr-un algoritm bcrypt cu un factor de "salt" ridicat în momentul salvării în baza de date. Serverul nu știe niciodată parola în clar.

6. Analiza Componentelor Complexe (WebGL, AI, State)

6.1. Sistemul de Analiză AI (LLM Integration)

Când un prospect completează chestionarul lung de digitalizare, platforma generează automat un raport inteligent.

// Arhitectura fluxului de generare Raport AI (Simplified)
export async function generateAIReport(leadId, answers) {
  // 1. Construim Prompt-ul folosind un Template din DB
  const systemPrompt = "Ești un consultant senior IT care analizează datele IMM-urilor...";
  const userData = formatAnswersForPrompt(answers);
  
  // 2. Apelăm OpenAI prin Vercel AI SDK (pentru edge streaming)
  const response = await generateObject({
    model: openai('gpt-4-turbo'),
    schema: z.object({
      score: z.number().min(0).max(100),
      vulnerabilities: z.array(z.string()),
      recommendedServices: z.array(z.string()),
      executiveSummary: z.string()
    }),
    prompt: userData
  });
  
  // 3. Salvăm ca JSONB în tabelul Lead
  await db.lead.update({
    where: { id: leadId },
    data: { aiReport: response.object, score: response.object.score }
  });
}

De ce JSON Schema (Zod)? LLM-urile (precum GPT-4) returnează implicit text liber (nestructurat). Obligând AI-ul să respecte un Zod schema, garantăm că răspunsul său va fi un obiect JSON perfect structurat (cu procente, liste bullet-point), care poate fi randat nativ de componentele noastre React în Dashboard, fără erori de parsare.

6.2. Componenta LiquidEther (WebGL Fluid Simulation)

Componenta LiquidEther din hero-landing.tsx nu este un simplu video, ci o simulare fizică a fluidelor calculată în timp real folosind placa video a utilizatorului. Aceasta implementează 3 programe Shader (GLSL):

  1. Advection Shader: Mută cerneala prin spațiu conform vectorilor de viteză.
  2. Poisson/Jacobi Solver: Calculează presiunea fluidului pentru a menține incompresibilitatea.
  3. Display Shader: Suprapune culorile noastre din paleta Floral Earth peste masa de fluid. Pentru compatibilitatea cu tema luminoasă (Ivory), a fost modificat canalul alpha (transparența) al shader-ului pentru a nu randa negru/alb, permițând fundalului de CSS să treacă prin fluid.

Optimizare de performanță: Componenta dispune de un mecanism IntersectionObserver. Simularea WebGL se oprește (pause) complet atunci când utilizatorul face scroll și secțiunea Hero iese din ecran, economisind bateria și procesorul pe dispozitivele mobile (evitând consumul inutil de GPU).

7. Strategie SEO și Optimizări de Performanță

Fiind o platformă care își propune să atragă organic IMM-uri interesate de digitalizare, SEO-ul a fost o prioritate arhitecturală:

7.1. Generarea Semantică a Metadatelor (Metadata API)

În Next.js 14, metadatele nu se mai introduc manual în tag-uri <head>, ci folosind API-ul de Metadata. Am implementat funcția constructMetadata() (în lib/utils.ts) care generează automat titluri, descrieri meta, tag-uri OpenGraph (pentru Facebook/LinkedIn) și tag-uri Twitter Cards pentru absolut fiecare pagină. Când un articol de blog este încărcat din baza de date, metadatele se adaptează dinamic pentru a include titlul și imaginea articolului.

7.2. JSON-LD (Schema.org Structured Data)

Pe homepage (app/(marketing)/page.tsx) am integrat scripturi application/ld+json invizibile pentru utilizator, dar citite de Google. Acestea declară VreauDigitalizare ca fiind o entitate de tip LocalBusiness și Organization, specificând locația (România), serviciile și linkurile sociale. Acest lucru crește radical șansele de a apărea în Google Knowledge Graph (caseta din dreapta căutărilor).

7.3. Optimizarea Imaginilor (next/image)

Nu folosim deloc tag-ul HTML standard <img>. Toate imaginile folosesc componenta Next.js <Image> care realizează: 1) Convertire automată la format WebP; 2) Redimensionare automată în funcție de dispozitiv; 3) Lazy-loading nativ (imaginea nu se încarcă până nu ajungi cu scroll-ul la ea), salvând lățime de bandă.

8. CI/CD, Deployment și Variabile de Mediu

Platforma este construită pentru a rula în cloud-ul serverless Vercel, conectată la o instanță de bază de date pe platforme precum Supabase, Neon sau Railway.

Procesul de Continuous Deployment (CI/CD)

Orice commit împins pe branch-ul main din GitHub declanșează automat următorul flux de lucru pe serverele Vercel:

  1. Clonare Repository: Vercel descarcă ultimul cod sursă.
  2. Install & Linting: Se instalează dependențele (npm install). Dacă TypeScript sau ESLint detectează erori (ex: un tip de date greșit în Prisma), build-ul se oprește și codul spart nu ajunge în producție.
  3. Prisma Generate: Se generează clientul de baze de date bazat pe schema curentă.
  4. Build: Next.js compilează paginile statice și asamblează bundle-urile JavaScript de client și server.
  5. Atomic Deployment: Noua versiune intră online instantaneu pe rețeaua globală (Edge Network), înlocuind versiunea veche fără nicio secundă de downtime (Zero Downtime Deployment).

Gestionarea Securizată a Variabilelor de Mediu

Codul aplicației folosește pachetul @t3-oss/env-nextjs pentru validarea riguroasă a variabilelor de mediu (fișierul .env). Când aplicația pornește, biblioteca Zod verifică dacă toate variabilele esențiale există:

  • DATABASE_URL: String-ul de conexiune PostgreSQL.
  • NEXTAUTH_SECRET: Cheia de criptare (JWT) pentru sesiuni. Fără ea, nimeni nu se poate autentifica.
  • OPENAI_API_KEY: Cheia necesară pentru generarea rapoartelor.
  • RESEND_API_KEY: Pentru trimiterea emailurilor de notificare automată.

Dacă vreuna din aceste variabile lipsește pe serverul de producție, aplicația va refuza să compileze, alertând echipa DevOps în loc să producă erori greu de detectat la utilizatorii finali.
